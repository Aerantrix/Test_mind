Testing methodology: recording and processing — Inter-Mind

Manual testing guide for Inter-Mind

This guide covers the methodology of testing: how to record, what to record with, where to report. The actual scenarios (what exactly to test) are in a separate document.

1. What to prepare in advance

Environment and URL

We test on production: https://intermind.com There are no other environments — everything you do happens on the real system. So do not enter real payment details unless explicitly told otherwise.

Accounts

You create all accounts yourself, for your own use. No one hands them to you in advance. The specific tasks that need checking are described in a separate document — this one only covers general technical rules.

  • A fresh email for every attempt that requires a “clean” user (signup, password reset, etc.). If you reuse an address, the system will say “already registered”.
  • Where to get fresh addresses: the Gmail plus trick — your.email+test1@gmail.com, +test2@gmail.com and so on. Gmail treats them as distinct, but the mail still arrives in your main inbox. An endless free source. As a last resort — disposable inboxes (mail.tm, 10minutemail), but their domains are often blocked in production.
  • Multiple participants at once — open 2+ browser sessions: a main window + a private window / a different browser / a different device. Or test with a live person on the other end.
  • Stripe test cards for payment checks. The live Stripe on production may not accept test numbers — confirm in advance whether a test payments environment is set up. If unsure, leave billing alone.
  • After the session, don’t delete the accounts you created right away. They’ll come in handy if a bug report requires you to go back and reproduce.

Browsers

Inter-Mind officially supports:

Browser Minimum
Chrome 90+
Edge 90+ (Chromium)
Safari 15+ (screen-sharing — 17+)
Firefox 100+
Opera, Brave, Arc Chromium 90+
iOS Safari current
Android Chrome current

Rules: - Browser version — always the latest stable. - Open in a normal window, not in incognito (permissions and cookies between sessions misbehave there). - Don’t use blocker extensions during the session — Inter-Mind explicitly warns about this in docs/browsers: NordVPN, Ghostery, tab-suspender plugins break WebRTC. - Before a demanding session, run browser-test — a 30-second self-check of mic/camera/network/codecs.

Hardware - A headset or headphones with a microphone. You can’t record without them — there will be echo. - Microphone close to your mouth, in a quiet room. - A second monitor helps — test on one, take notes on the other.

macOS system permissions (System Settings → Privacy & Security): - ✅ Screen Recording — for QuickTime / Screenshot - ✅ Microphone — for the browser and for QuickTime - ✅ Camera — for the browser

2. How to set up recording with sound

We need to record three things at once: 1. The screen — what you see in the browser and the system 2. Your voice (microphone) — you narrate your actions out loud 3. The other participants’ audio (system sound) — otherwise it’s unclear what was said “off-screen”

By default macOS does not record system audio. So we install a free “virtual audio cable” — BlackHole.

Step 1. Install BlackHole

In the terminal:

brew install blackhole-2ch

If you don’t have brew — install it from brew.sh.

Step 2. Create a Multi-Output Device (to both hear and record)

  1. Open “Audio MIDI Setup” (/Applications/Utilities/Audio MIDI Setup.app or via Spotlight: Audio MIDI).
  2. Bottom-left + → “Create Multi-Output Device”.
  3. Check the boxes for:
    • your headphones (or speakers)
    • BlackHole 2ch
  4. Name it, for example, Speakers + BlackHole.

Step 3. Switch the system output

System Settings → Sound → Output → pick Speakers + BlackHole. Now all system audio (the voices of conference participants) goes both to your headphones and to BlackHole — where OBS will pick it up.

After the session, switch the output back to your normal headphones, otherwise the next meeting will sound “off”.

3. How to record a session (OBS)

We use OBS Studio — the only method. It records video plus two separate audio tracks (microphone and system sound) into one file. This is critical: during processing we split them apart and figure out who said what.

Step 1. Install OBS

brew install --cask obs

Or download from obsproject.com (free).

On first launch OBS will ask for screen recording permission — grant it (System Settings → Privacy → Screen Recording).

Step 2. Set up the scene (once)

In an empty scene, click + under “Sources” and add, one by one:

  1. macOS Screen Capture (or “Screen Capture”) → pick the main display.
  2. Audio Input Capture → device: your microphone. Name it Mic.
  3. Audio Input Capture → device: BlackHole 2ch. Name it System.

Step 3. Split audio across tracks

This is the key step — without it, both tracks will get mixed together.

  1. In the “Audio Mixer” (bottom of OBS), each track has a ⚙ icon → “Advanced Audio Properties”.
  2. Find the Mic row → in the “Tracks” column leave only the checkbox on 1.
  3. Find the System row → leave only the checkbox on 2.
  4. Close.

Step 4. Configure recording

Settings → Output: - Output Mode: Simple (good enough). - Recording Quality: High Quality, Medium File Size. - Recording Format: MP4 (or MKV — safer if OBS crashes, but you’ll then need to “remux” into MP4 via File → Remux). - Encoder: Apple VT H264 Hardware (fast, doesn’t stress the machine). - Audio Track: tick both — 1 and 2.

Settings → Audio: - Sample Rate: 48 kHz.

Step 5. Record

  1. Click “Start Recording” in OBS.
  2. Minimize OBS, switch to Chrome / Inter-Mind, start testing.
  3. When done — “Stop Recording” in OBS.
  4. The file lands in the folder set under Settings → Output → Recording Path (by default ~/Movies).

The resulting .mp4 will contain two independent audio tracks. By default QuickTime plays only the first one (your voice) — that is normal, the file is fine. To hear track 2 in QuickTime: Window → Show Audio Tracks.

How to verify that the file really has 2 tracks

ffprobe -hide_banner recording.mp4 2>&1 | grep -i audio

You should see two Audio: lines. If there’s only one — the audio track configuration in OBS didn’t stick, double-check Step 3.

Length

  • 15–30 minutes per session. Longer and you get tired, you start forgetting to narrate.
  • If the task is large — split it into several recordings and tag the filenames part1, part2.

What to say while recording

This is the most important part of the whole test. You don’t need to write anything down by hand during the session — everything goes through your voice. The written report is built afterwards from the transcript of what you say. A silent screencast is useless.

Narrate out loud what you’re doing, what you expect, how it feels:

“Now I want to invite a new participant… I’m looking for the button… I don’t see it at the top… I went into the profile menu, not there either… I went into team settings, still not there… ah, turns out it’s inside the participants list, which is not obvious…”

To make the later sorting easier, explicitly call out the type of issue. Use three keywords:

Word When to say it Example
“doesn’t work” / “bug” When something is genuinely broken — an error, wrong behaviour “Doesn’t work — I clicked ‘Send’, a minute passed, the email never arrived”
“awkward” When it works, but takes extra steps or is annoying “Awkward — to invite a person it takes 4 clicks, the button is buried”
“unclear” When you couldn’t guess, couldn’t figure it out, the wording is opaque “Unclear — what does the ‘Demo TTL’ label on the home screen mean?”

Not necessary every single time. But at least once at the start of each observation — the processing pipeline keys off these words to file everything into the right buckets.

One more thing: name what you see on screen, in words. “I’m now clicking the blue ‘Create meeting’ button in the top right corner” is better than just a click. That way the transcript can be matched up with video frames even without a cursor.

Narrate absolutely everything, including successful actions

This matters: not just the problems. Every click, every step, every successful action — narrate it out loud.

Why:

  • Context for the problems. To know a bug is at step 5, we need to know steps 1–4 went fine. Silence is ambiguous: either everything was OK, or you forgot to mention a problem.
  • Scenario coverage. The transcript shows which steps you actually went through and which you skipped. Without narration it looks like everything was tested, when in fact only half was.
  • Indirect UX signals. “Created a meeting — it worked first try, everything was intuitive” is also valuable information; it confirms the happy path works.
  • Speed and smoothness. “Slow to load, around 6 seconds” or “I had to wait for the button to appear” isn’t a bug and isn’t an ‘unclear’ either, but it’s an important signal for the team.

What a good commentary sounds like:

“Opening intermind.com… home page loaded in a second, looks fine… clicking ‘Sign up’ at the top right… the signup form opened, Email and Password fields, all standard… entering email, entering password, clicking ‘Continue’… confirmation email arrived in 10 seconds, all good… clicking the link…”

Compare that to a silent video of the same actions — there’s nothing to analyse. If something is happening on screen and you’re silent — it’s as if that minute didn’t happen.

Rule: “Recorded but stayed silent” = “didn’t record”.

4. Where to report bugs

Functional bugs — the built-in “Report a Bug” button

If you catch something genuinely broken (an error, a non-working feature) — immediately hit the “Report a Bug” button inside the app itself.

What to do: 1. Click “Report a Bug” right in the interface. 2. Describe in one or two sentences what happened. 3. Send. 4. Say out loud into the microphone: “sent a bug report via the button” — so the transcript later shows that this issue is already in Sentry.

What gets attached automatically: - The session video-replay for the last few minutes (Sentry Session Replay) - The error stack, app version, your email - An event in PostHog

UX observations — voice only

If something works but is awkward/unclear — do not use the “Report a Bug” button. There’s no error there, Sentry won’t accept it as a bug. Just say it out loud into the microphone: “awkward — …” or “didn’t get it — …”. That’s enough. A written log of these will be assembled automatically during processing.

5. What the final output looks like (for context, not for you to fill in)

From your video, a report like the one below is assembled automatically. You don’t need to write it — it gets built from the transcript of what you say plus video frames. I’m showing it just so you understand the format you’re speaking into:

# UX report: <feature or area>
Tester: <name>
Date: 2026-05-18
Video: session-2026-05-18.mp4
Browser / OS: Chrome 124 / macOS 15.2

---

## Doesn’t work (functional bugs)

| Timestamp | What happened | In Sentry? |
|-----------|---------------|------------|
| 03:14     | Clicked “Invite”, email never arrived after 5 minutes | yes |
| 12:08     | Shared screen — the other participant saw a black square | yes |

## Awkward

| Timestamp | Description |
|-----------|-------------|
| 05:40     | Inviting a participant takes 4 clicks — the button is buried in a submenu |
| 18:22     | After payment there’s no confirmation that the subscription is active |

## Unclear

| Timestamp | Description |
|-----------|-------------|
| 02:10     | The “Demo TTL” label — unclear what it means |
| 09:55     | The “🔴” icon — I read it as “offline”, it actually means “recording in progress” |
| 21:30     | The error text “Failed to fetch” — doesn’t explain what to do |

The clearer you say “doesn’t work / awkward / unclear” in the video, the more accurate this table will be without any manual intervention.

6. After the session — how to hand in your results

Into a single folder named session-YYYY-MM-DD-your_name/ put:

  1. screencast.mp4 — the video recording with two audio tracks (or several partN.mp4)
  2. notes.md — if after the session you remembered something important that you didn’t get to say out loud (optional, usually not needed)

Where to send it: Google Drive / a shared folder (we’ll arrange the details separately).

What you don’t need to do: - Keep a parallel written log during the session — it’s all voice. - Take separate screenshots — they’re already in the video. - Transcribe the audio by hand — that’s automated. - Write up descriptions of bugs you already filed via the “Report a Bug” button — they’re in Sentry, and you said out loud that you sent them.

7. Possible issues

Only one audio track in the file → In OBS → “Advanced Audio Properties” → Mic should have only checkbox “1” ticked, System — only “2”. And in Settings → Output → Audio Track both must be ticked.

My voice isn’t in the screencast (track 1 is silent) → In OBS, the Mic source points at the wrong device (e.g. BlackHole instead of the actual microphone). Recheck the source properties.

Participants’ voices aren’t in the screencast (track 2 is silent) → System audio is bypassing BlackHole. Check: System Settings → Sound → Output = Speakers + BlackHole before the conference starts.

I can’t hear participants in my headphones while recording → In the Multi-Output Device you forgot to tick the headphones. Open Audio MIDI Setup and check the box.

OBS isn’t recording BlackHole audio → On macOS 14+ you need to grant OBS microphone permission (even though BlackHole is “system audio”, macOS sees it as an input): System Settings → Privacy → Microphone → OBS ✅.

The recording stopped on its own → Disk ran out of space, or the Mac went to sleep. Before the session: caffeinate -d & in the terminal (the Mac won’t sleep while the command is running).

The file is big (>1 GB for 30 minutes) → That’s normal for full HD. Don’t compress it manually — you’ll lose the quality needed to recognize on-screen text.

The browser asks for permissions in the middle of the recording → Grant microphone/camera/notifications in advance, before starting OBS.

QuickTime plays only one track → That’s normal. The file is fine, both tracks are inside. Check with ffprobe recording.mp4 2>&1 | grep Audio.
