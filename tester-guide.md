# Инструкция для ручной проверки Inter-Mind

> Это инструкция про **технологию** тестирования: как писать, чем писать, куда репортить. Конкретные сценарии (что именно тестировать) — в отдельном документе.

---

## 1. Что нужно подготовить заранее

**Среда и URL**

Тестируем **на проде**: **https://intermind.com**
Других сред нет — всё, что ты делаешь, происходит на реальной системе. Поэтому **не вводи реальные платёжные данные**, если иное не оговорено отдельно.

**Аккаунты**

Все аккаунты ты заводишь **сам**, под себя. Никто их тебе заранее не выдаёт. Конкретные задачи, которые требуют проверки, описаны в отдельном документе — здесь только общие технические правила.

- **Новый email на каждую попытку, требующую "чистого" пользователя** (регистрация, восстановление пароля и т.п.). Если переиспользовать адрес, система ответит «уже зарегистрирован».
- **Где брать свежие адреса:** Gmail-трюк с плюсом — `твой.email+test1@gmail.com`, `+test2@gmail.com` и т.д. Gmail считает их разными, письма всё равно приходят на основной ящик. Бесконечный бесплатный источник. На крайний случай — временные ящики (mail.tm, 10minutemail), но их домены часто блокируются на проде.
- **Несколько участников одновременно** — открывай 2+ браузерных сессии: основное окно + приватное окно / другой браузер / другое устройство. Либо тестируй с живым человеком на другой стороне.
- **Тестовые карты Stripe** для проверок оплаты. На проде боевой Stripe может не принять тестовые номера — уточни заранее, заведена ли тестовая среда оплат. Если не уверен — биллинг не трогай.
- **После сессии не удаляй** созданные учётки сразу. Они пригодятся, если по баг-репорту нужно будет вернуться и воспроизвести.

**Браузеры**

Inter-Mind официально поддерживает:

| Браузер | Минимум |
|---|---|
| Chrome | 90+ |
| Edge | 90+ (Chromium) |
| Safari | 15+ (screen-sharing — 17+) |
| Firefox | 100+ |
| Opera, Brave, Arc | Chromium 90+ |
| iOS Safari | актуальная |
| Android Chrome | актуальная |

**Правила:**
- Версия браузера — всегда **последняя стабильная**. 
- Открывай в **обычном окне**, не в инкогнито (там вылетают разрешения и куки между сессиями).
- Не используй расширения-блокировщики во время сессии — Inter-Mind на этот счёт даёт явное предупреждение в [docs/browsers](https://intermind.com/en/troubleshooting/browsers): NordVPN, Ghostery, suspender-плагины ломают WebRTC. 
- Перед сложной сессией прогони [browser-test](https://intermind.com/browser-test) — 30-секундная самопроверка микрофона/камеры/сети/кодеков.

**Оборудование**
- **Гарнитура или наушники с микрофоном.** Записывать без них нельзя — будет эхо.
- Микрофон близко ко рту, в тихой комнате.
- Хорошо, если есть второй монитор — на одном тестируешь, на другом ведёшь заметки.

**Системные разрешения macOS** (`Системные настройки → Конфиденциальность и безопасность`):
- ✅ Запись экрана — для QuickTime / Screenshot
- ✅ Микрофон — для браузера и для QuickTime
- ✅ Камера — для браузера

## 2. Как настроить запись со звуком

Нам нужно записывать одновременно:
1. **Экран** — что видно в браузере и системе
2. **Твой голос** (микрофон) — ты вслух комментируешь действия
3. **Звук собеседников** (системный звук) — иначе непонятно, что было сказано «за экраном»

macOS по умолчанию **не пишет системный звук**. Поэтому ставим бесплатный «виртуальный аудио-кабель» — **BlackHole**.

### Шаг 1. Установить BlackHole

В терминале:
```bash
brew install blackhole-2ch
```
Если нет `brew` — установи с [brew.sh](https://brew.sh).

### Шаг 2. Создать Multi-Output Device (чтобы и слышать, и писать)

1. Открой **«Настройка Audio-MIDI»** (`/Applications/Utilities/Audio MIDI Setup.app` или через Spotlight: `Audio MIDI`).
2. Слева снизу `+` → **«Создать многовыходное устройство»**.
3. Поставь галочки:
   - твои **наушники** (или динамики)
   - **BlackHole 2ch**
4. Назови, например, `Speakers + BlackHole`.

### Шаг 3. Переключить системный выход

`Системные настройки → Звук → Выход` → выбери **`Speakers + BlackHole`**.
Теперь весь звук системы (голоса участников конференции) идёт и тебе в наушники, и в BlackHole — откуда его подхватит OBS.

После сессии верни выход на обычные наушники, иначе следующая встреча будет звучать «не туда».

## 3. Как записывать сессию (OBS)

Используем **OBS Studio** — единственный метод. Он пишет видео + **две отдельные аудио-дорожки** (микрофон и системный звук) в один файл. Это критично: на обработке мы их разделим и поймём, кто что сказал.

### Шаг 1. Установить OBS

```bash
brew install --cask obs
```
Или скачай с [obsproject.com](https://obsproject.com) (бесплатно).

При первом запуске OBS попросит разрешение на запись экрана — **выдай его** (`Системные настройки → Конфиденциальность → Запись экрана`).

### Шаг 2. Настроить сцену (один раз)

В пустой сцене жми `+` под «Источники» и добавь по очереди:

1. **`macOS Screen Capture`** (или «Захват экрана») → выбери основной дисплей.
2. **`Audio Input Capture`** → устройство: **твой микрофон**. Назови `Mic`.
3. **`Audio Input Capture`** → устройство: **BlackHole 2ch**. Назови `System`.

### Шаг 3. Разнести аудио по дорожкам

Это ключевой шаг — без него обе дорожки смешаются.

1. В «Микшере аудио» (внизу OBS) у каждой дорожки есть иконка ⚙ → **«Расширенные параметры аудио»**.
2. Найди строку `Mic` → в столбце «Дорожки» оставь галочку только на **1**.
3. Найди строку `System` → оставь галочку только на **2**.
4. Закрой.

### Шаг 4. Настроить запись

`Настройки → Вывод`:
- Режим вывода: **Простой** (хватает).
- Качество записи: **Высокое качество, средний размер**.
- Формат записи: **MP4** (или **MKV** — безопаснее при сбоях, но потом нужно «remux» в MP4 через `Файл → Remux`).
- **Кодировщик: Apple VT H264 Hardware** (быстро, не нагружает машину).
- **Аудиодорожка:** отметь **обе** — 1 и 2.

`Настройки → Аудио`:
- Частота дискретизации: **48 kHz**.

### Шаг 5. Записать

1. Жми **«Начать запись»** в OBS.
2. Сворачивай OBS, иди в Chrome / Inter-Mind, тестируй.
3. По окончании — **«Остановить запись»** в OBS.
4. Файл лежит в папке, указанной в `Настройки → Вывод → Путь записи` (по умолчанию `~/Movies`).

В получившемся `.mp4` будут две независимые аудио-дорожки. QuickTime по умолчанию играет только первую (твой голос) — это **нормально**, файл рабочий. Чтобы услышать дорожку 2 в QuickTime: `Окно → Показать звуковые дорожки`.

### Как проверить, что в файле действительно 2 дорожки

```bash
ffprobe -hide_banner recording.mp4 2>&1 | grep -i audio
```
Должно быть две строчки `Audio:`. Если одна — настройка аудио-дорожек в OBS не сохранилась, перепроверь Шаг 3.

### Длительность

- **15–30 минут на одну сессию.** Дольше — устаёшь, начинаешь забывать комментировать.
- Если задача большая — разбей на несколько записей, в названии укажи `part1`, `part2`.

### Что говорить во время записи

**Это самая важная часть всего теста. Ничего руками во время сессии записывать не надо — всё голосом.** Текстовый отчёт построится потом, из транскрипта твоей речи. Молчаливый скринкаст бесполезен.

Говори вслух **что делаешь, что ожидаешь, что чувствуешь**:

> «Сейчас я хочу пригласить нового участника… ищу кнопку… не вижу её сверху… полез в меню профиля, тоже нет… полез в настройки команды, тоже нет… ага, оказывается она внутри списка участников, это неочевидно…»

Чтобы потом легче было раскладывать по полкам, **проговаривай тип проблемы явно**. Используй три ключевых слова:

| Слово | Когда говорить | Пример |
|---|---|---|
| **«не работает»** / «баг» | Когда что-то реально сломано, ошибка, не то поведение | «Не работает — нажал „Отправить“, прошла минута, письмо не пришло» |
| **«неудобно»** | Когда работает, но требует лишних действий или раздражает | «Неудобно — чтобы пригласить человека, нужно 4 клика, кнопка спрятана» |
| **«не понятно»** | Когда не догадался, не разобрался, текст невнятный | «Не понятно — что значит надпись „Demo TTL“ на главной?» |

Не обязательно каждый раз. Но **хотя бы раз в начале каждого замечания** — обработка по этим словам разложит всё в правильные категории.

Ещё одно: **называй то, что видишь на экране словами**. «Сейчас нажимаю синюю кнопку „Создать встречу“ в верхнем правом углу» лучше, чем просто клик. Тогда транскрипт можно сопоставить с кадрами видео даже без курсора.

### Комментируй абсолютно всё, в том числе успешные действия

Это важно: **не только проблемы**. Каждый клик, каждый шаг, каждое успешное действие — проговаривай вслух.

Почему:

- **Контекст для проблем.** Чтобы понять, что баг на шаге 5, нужно знать, что шаги 1–4 прошли нормально. Молчание двусмысленно: то ли всё ок, то ли ты забыл сказать про проблему.
- **Покрытие сценария.** По транскрипту видно, какие шаги ты реально прошёл, а какие пропустил. Без озвучки кажется, что протестировано всё, а на деле — половина.
- **Косвенные UX-сигналы.** «Создал встречу — получилось с первой попытки, всё интуитивно» — это тоже ценная информация, она подтверждает, что happy path работает.
- **Скорость и плавность.** «Загружается долго, секунд 6» или «пришлось ждать, пока появится кнопка» — это не баг и не «непонятно», но это важный сигнал для команды.

Как звучит правильный комментарий:

> «Открываю intermind.com… главная загрузилась за секунду, выглядит нормально… жму „Sign up“ справа сверху… открылась форма регистрации, поля Email и Password, всё стандартно… ввожу email, ввожу пароль, жму „Continue“… пришло письмо с подтверждением за 10 секунд, всё хорошо… перехожу по ссылке… »

Сравни с молчаливым видео тех же действий — там нет ничего, что можно проанализировать. **Если в кадре что-то происходит, а ты молчишь — этой минуты как будто не было.**

Правило: **«Снят, но молчал» = «не снимал».**

## 4. Куда репортить баги

### Функциональные баги — встроенная кнопка «Report a Bug»

Если поймал что-то **реально сломанное** (ошибка, неработающая функция) — **сразу** жми кнопку «Report a Bug» в самом приложении.

**Что сделать:**
1. Жми «Report a Bug» прямо в интерфейсе.
2. Опиши **одной-двумя фразами**, что произошло.
3. Отправь.
4. **Проговори вслух** в микрофон: «отправил баг-репорт через кнопку» — чтобы потом по транскрипту было видно, что эта проблема уже в Sentry.

**Что прицепится автоматически:**
- Видео-replay сессии за последние минуты (Sentry Session Replay)
- Стек ошибки, версия приложения, твой email
- Событие в PostHog

### UX-замечания — только голосом

Если что-то *работает, но неудобно/непонятно* — кнопку «Report a Bug» **не используй**. Там нет ошибки, Sentry это не примет за баг.
Просто **проговори вслух** в микрофон: «неудобно — …» или «не понял — …». Этого достаточно. Текстовый лог из этого будет собран автоматически на этапе обработки.

## 5. Что получится в итоге (для понимания, не для заполнения)

Из твоего видео автоматически соберётся вот такой отчёт. Тебе **не нужно** его писать — он строится из транскрипта твоей речи + кадров видео. Показываю просто чтобы ты понимал, под какой формат говорить:

```markdown
# UX-отчёт: <фича или область>
Тестировщик: <имя>
Дата: 2026-05-18
Видео: session-2026-05-18.mp4
Браузер / OS: Chrome 124 / macOS 15.2

---

## Не работает (функциональные баги)

| Таймкод | Что произошло | В Sentry? |
|---------|---------------|-----------|
| 03:14   | Нажал «Пригласить», письмо не пришло за 5 минут | да |
| 12:08   | Расшарил экран — у второго участника чёрный квадрат | да |

## Неудобно

| Таймкод | Описание |
|---------|----------|
| 05:40   | Чтобы пригласить участника, нужно 4 клика — кнопка спрятана в подменю |
| 18:22   | После оплаты нет подтверждения, что подписка активна |

## Непонятно

| Таймкод | Описание |
|---------|----------|
| 02:10   | Надпись «Demo TTL» — непонятно, что значит |
| 09:55   | Иконка «🔴» — принял за «offline», оказалось «идёт запись» |
| 21:30   | Текст ошибки «Failed to fetch» — не объясняет, что делать |
```

Чем чётче в видео ты проговариваешь «не работает / неудобно / не понятно» — тем точнее эта таблица получится без ручного вмешательства.

## 6. После сессии — как сдать результат

В одну папку с именем `session-YYYY-MM-DD-твоё_имя/` сложи:

1. `screencast.mp4` — видеозапись с двумя аудио-дорожками (или несколько `partN.mp4`)
2. `notes.md` — если уже после сессии вспомнил что-то важное, чего не успел сказать вслух (опционально, обычно не нужно)

**Куда передать:** Google Drive / общая папка (договоримся отдельно).

**Не нужно:**
- Вести параллельный текстовый лог во время сессии — всё голосом.
- Делать скриншоты отдельно — они уже в видео.
- Расшифровывать аудио вручную — это автоматически.
- Писать описания багов, которые ты уже отправил кнопкой «Report a Bug» — они в Sentry, и ты вслух сказал, что отправил.

## 7. Возможные проблемы

**В файле только одна аудио-дорожка**
→ В OBS → «Расширенные параметры аудио» → у `Mic` должна стоять только галочка «1», у `System` — только «2». И в `Настройки → Вывод → Аудиодорожка` отмечены **обе**.

**В скринкасте нет моего голоса (дорожка 1 тихая)**
→ В OBS источник `Mic` указывает на ошибочное устройство (например, на BlackHole вместо реального микрофона). Перепроверь в свойствах источника.

**В скринкасте нет голосов участников (дорожка 2 тихая)**
→ Системный звук идёт мимо BlackHole. Проверь: `Системные настройки → Звук → Выход` = `Speakers + BlackHole` **до начала** конференции.

**Не слышу собеседников в наушниках во время записи**
→ В Multi-Output Device забыл отметить наушники. Открой Audio-MIDI Setup, поставь галочку.

**OBS не пишет звук BlackHole**
→ В macOS 14+ нужно дать OBS разрешение на микрофон (даже если BlackHole — это «системный звук», macOS видит его как input): `Системные настройки → Конфиденциальность → Микрофон → OBS ✅`.

**Запись остановилась сама**
→ Кончилось место на диске или Mac ушёл в сон. Перед сессией: `caffeinate -d &` в терминале (Mac не уснёт, пока команда работает).

**Размер файла большой (>1 GB за 30 мин)**
→ Это нормально для full HD. Не сжимай вручную — теряется качество распознавания текста на экране.

**Браузер просит разрешения посреди записи**
→ Раздай микрофон/камеру/уведомления заранее, до старта OBS.

**QuickTime играет только одну дорожку**
→ Это нормально. Файл рабочий, обе дорожки внутри. Проверь через `ffprobe recording.mp4 2>&1 | grep Audio`.

## 8. Чек-лист перед стартом сессии

- [ ] Гарнитура подключена, микрофон проверен
- [ ] `Системные настройки → Звук → Выход` = `Speakers + BlackHole`
- [ ] OBS открыт, сцена настроена (Mic → дорожка 1, System → дорожка 2)
- [ ] В `Настройки → Вывод → Аудиодорожка` отмечены **обе** (1 и 2)
- [ ] Браузер открыт в обычном окне, не в инкогнито
- [ ] Свободное место на диске ≥ 5 ГБ
- [ ] `caffeinate -d &` запущен, Mac не уснёт
- [ ] Уведомления macOS в режиме «Не беспокоить» (`Ctrl + ⌥ + Z` в Mission Control)
- [ ] Настроился говорить вслух «не работает / неудобно / не понятно» при каждом замечании
- [ ] Тестовая 30-секундная запись сделана и проверена `ffprobe` (две дорожки на месте)
